# My Picks — Family Deal Curation — Product Canvas

## Stage 1: Thesis

### Value Proposition (one sentence)

For **family members browsing deals between calls**, My Picks lets them star deals they find interesting, creating a personal shortlist that the operator can see and optionally pull into the Call Prep.

### Audience Scope

Scope: **Operator**

Would I use this *this week*? Yes — Sunday call. Family is already browsing. They need a way to signal "I found something."

### The Problem

The operator curates the Call Prep (top-down), but family members have no way to surface deals they like (bottom-up). On Sunday's call, if Mom found a deal she likes while browsing, she has no way to reference it quickly. She'd have to scroll through the review list and say "it was the one in Alabama..." The deal sourcing is one-directional — operator → family. It should be bidirectional.

### The Bet

If family members can star deals with one tap, two things happen: (1) they engage more because saving feels like participation, not just reading, and (2) the operator gets signal about what the family cares about, improving Call Prep curation.

### Conviction check

Operator scope: passes. Family is already browsing on prod. The engagement signal is needed before Sunday.

## Stage 2: Shape

### One-Paragraph PR/FAQ

Family members can now star deals they find interesting while browsing. Stars accumulate in a "My Picks" tab on the Review page. The operator sees a "Family Picks" tab showing what the family has starred, with a count per deal. When curating the Call Prep, deals with family stars surface as signals. One tap to save, one tap to unsave. No workflow, no proposal process — just "I liked this."

### Users / Personas

1. **Family members (Collaborators)** — browse Review, star deals, reference their picks on the call
2. **Sanjay (Operator)** — sees family picks as curation signal, pulls starred deals into Call Prep
3. **Mom (Viewer, 76)** — needs one-tap star, large tap target, no confusion

### Access Scope

- [x] Trusted circle (family — authenticated)
- All authenticated users can star deals and see their own picks
- Operator sees aggregated family picks

### Key Outcomes

1. **Family engagement increases** — starring is lower friction than voting. More participation before the call.
2. **Operator gets bottom-up signal** — "3 family members starred this" informs Call Prep curation.
3. **Call prep becomes bidirectional** — family surfaces deals the operator might miss.

## Stage 3: Commit

Operator scope — skip full PR/FAQ.

### Risks & Assumptions

| Risk | Mitigation |
|------|-----------|
| Family doesn't use it | Star is one-tap, zero friction. Worse case: unused feature, no harm. |
| Too many features confuse family | Picks live within Review (no new nav). Star icon is universally understood. |
| Privacy concern (others see my picks) | Operator sees aggregated. Family sees their own picks only. No one sees who starred what (except operator). |

### Constraints

- **Deadline**: Functional for Sunday call
- **Minimal model**: DealFavorite (dealId + userId + createdAt). No status, no workflow.
- **No new nav items**: Tab within Review page

### What's in scope

1. Star icon on deal rows (Review list) and deal detail pages — one tap save/unsave
2. "My Picks" tab on Review page — filtered to your starred deals
3. "Family Picks" tab on Review page (operator only) — deals starred by any family member, count per deal
4. Star count visible on deal rows: "★ 3" when family members have starred
5. Call Prep "Add Opportunity" search shows star count as signal

### What's deferred

| Feature | When |
|---------|------|
| Leaderboard ("Mom saved 3 · Rasik saved 5") | Phase 2 — after we see engagement patterns |
| "Propose for call" formal workflow | Phase 2 — if family wants more structure |
| Star from digest email | Phase 2 — needs magic link integration |
| Star from Call Prep brief | Phase 2 |

### Data model

```prisma
model DealFavorite {
  id        String   @id @default(cuid())
  dealId    String
  deal      Deal     @relation(fields: [dealId], references: [id], onDelete: Cascade)
  userId    String
  user      User     @relation(fields: [userId], references: [id])
  createdAt DateTime @default(now())

  @@unique([dealId, userId])
  @@index([userId])
  @@index([dealId])
}
```
